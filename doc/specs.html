Hi Paul,

Been thinking about the globe problem.  Here's what I've come up with so far.

MOVIE VS BITMAP

The first issue to resolve is whether the globe is rendered as a movie or by a bitmap-driven ActionScript program.

The globe could be delivered as a movie.  Each frame of the movie corresponds to a position of the globe and is rendered in full detail at build time.  There is a script that controls the motion and position of the movie - modifying playback speed to show acceleration and seeking to the correct frame at times when the globe deviates from straightahead spinning.

The globe could be rendered programatically.  Feed it a bitmap, and it renders as a globe, and animates.   Remember learning about Mercator Projections in school?  Never thought I'd be interested in them on the job.

There are some advantages to the movie approach.  One is that it will enable a realistic, detailed graphic.  We could even use photos of an actual globe.  Another is that it leverages the power of the Flash timeline, which was built for just this sort of thing.

There are also disadvantages to the movie approach.  One is file size.  A detailed movie of a spinning globe is likely to be large.  I don't have solid numbers to support this, just a hunch.  Larger file means longer startup latency, of course.  Another disadvantage is programmability.  A fully rendered movie will rule out the ability to modify the surface of the globe at run time.  Being able to switch among various views of the world: political, geographic, population, etc. at run
time seems like the basis for a lot of cool features.  Of course, to change the map at all would require a complete rebuild of the movie.  I'm also thinking that to render the globe spinning at different speeds might eventually call for a variable blurring effect, which requires run time bitmap manipulation.   This is not to say that a movie makes programming impossible, only that certain features are much more difficult.

Rendering the globe from a bitmap addresses the programmability concerns, but comes with its own problems.  Doing a precise mapping of flat bitmap to sphere is too computationally intense to pull off, especially with animation in the picture.  Here is a compromise that I think will work and look good:
1) Render the globe as a polyhedron, using Flash's fast bimap scaling/skewing.  Possibly leverage an open source library, such as: http://www.papervision3d.org/
2) Add light/shadow, background and a smooth edge for the globe (which polyhedrons lack) as a semi-transparent bitmap overlay.

So it comes down to a trade-off between image quality on one side and programmability on the other.  Which is more important to you?  I suspect the latter, but have to ask.  I have some thoughts about how various applications built around the globe would be authored, but they depend on the movie vs bitmap question.

IS FLASH 9 OK?

Another issue for you.  Flash ActionScript 3 is far superior to ActionScript 2, but requires Flash Player version 9, which has not yet reached 95% adoption.  For recent player version penetration numbers, check out: http://www.adobe.com/products/player_census/flashplayer/version_penetration.html 

It's common practice to do Flash version detection and point users who need to upgrade to the Flash upgrade page. 

Let me know whether relying on Flash 9 / ActionScript 3 is acceptable.
 
I'm at this conference in Seattle the first half of this week: http://www.360flex.com   Expecting to have some time to work on the globe problem while here, and there will be plenty of opportunities to get helpful ideas from other Flash developers! 

This could grow into something very cool, and I'm really happy to be involved in getting it going.  More as it happens...

-Jim
